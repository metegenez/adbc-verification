# Phase 2 Plan 01: TPC-H SF1 Data Loading and 22-Query Corpus Summary

**One-liner:** Pure Python TPC-H SF1 generator (random.seed(42)) producing 8 CSV files loaded via PostgreSQL COPY and MySQL LOAD DATA INFILE, plus all 22 TPC-H queries adapted for StarRocks ADBC passthrough.

## What Was Built

### Task 1: SF1 Data Generator (commit d4d5dc4)

`docker/generate-sf1-data.py` — a pure Python 3 stdlib script that generates all 8 TPC-H SF1 tables as CSV files to `docker/data/sf1/`:

| Table    | Rows       | File Size |
|----------|-----------|-----------|
| region   | 5          | <0.1 MB   |
| nation   | 25         | <0.1 MB   |
| supplier | 10,000     | 0.9 MB    |
| part     | 200,000    | 22.2 MB   |
| partsupp | 800,000    | 59.4 MB   |
| customer | 150,000    | 16.0 MB   |
| orders   | 1,500,000  | 126.5 MB  |
| lineitem | 6,000,121  | 677.7 MB  |

- Fixed seed `random.seed(42)` for deterministic, reproducible output
- Row counts validated at exit with non-zero exit on mismatch
- CSV headers match exact schema column names (verified against both PG and MySQL schemas)
- partsupp FK formula: TPC-H spec `((p + i*(S/4 + (p-1)/S)) mod S) + 1` giving 4 distinct suppliers per part
- CSVs excluded from git (`docker/data/sf1/*.csv` in `.gitignore`); directory tracked via `.gitkeep`

### Task 2: SF1 Init Scripts + Docker Compose (commit c934078)

**PostgreSQL (`docker/init/postgres/02-data.sql`):** TRUNCATE CASCADE + `\COPY` for all 8 TPC-H tables from `/sf1-data/` volume mount. Load order respects FK hierarchy.

**PostgreSQL (`docker/init/postgres/03-seed.sql`):** New file — preserves `test_data` (3 rows) and `departments` (3 rows) that Phase 1 tests depend on. These were extracted from the old `02-data.sql`.

**MySQL (`docker/init/mysql/02-data.sql`):** `SET FOREIGN_KEY_CHECKS=0` + TRUNCATE + `LOAD DATA INFILE` for all 8 TPC-H tables from `/var/lib/mysql-files/` (MySQL `secure_file_priv` default).

**MySQL (`docker/init/mysql/03-cross-join.sql`):** Extended to include `test_data` seed rows alongside existing `departments` seed.

**Docker Compose (`docker/docker-compose.yml`):**
- `sr-postgres`: added `./data/sf1/:/sf1-data/:ro` volume
- `sr-mysql`: added `./data/sf1/:/var/lib/mysql-files/:ro` volume
- Both backends: healthcheck `start_period` extended to 300s for SF1 load time

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 2 - Missing Critical Config] MySQL mount path changed to /var/lib/mysql-files/**
- **Found during:** Task 2
- **Issue:** Plan specified mount to `/sf1-data/` for MySQL. MySQL 8.0 `LOAD DATA INFILE` enforces `secure_file_priv` which defaults to `/var/lib/mysql-files/`. Mounting to `/sf1-data/` would cause all `LOAD DATA INFILE` statements to fail with `ERROR 1290 (HY000): --secure-file-priv option`.
- **Fix:** Changed Docker Compose MySQL volume mount to `./data/sf1/:/var/lib/mysql-files/:ro` and updated all `LOAD DATA INFILE` paths accordingly.
- **Files modified:** `docker/docker-compose.yml`, `docker/init/mysql/02-data.sql`
- **Commits:** c934078

**2. [Rule 2 - Missing Critical Config] Healthcheck start_period extended to 300s**
- **Found during:** Task 2
- **Issue:** Plan noted risk of init timeout but didn't specify an extended `start_period`. PostgreSQL's healthcheck had no `start_period` (defaulting to 0s), which means `service_healthy` would time out before SF1 loading completes (~60s for PostgreSQL at bulk COPY speeds, ~120s for MySQL).
- **Fix:** Added `start_period: 300s` to both `sr-postgres` and `sr-mysql` healthchecks.
- **Files modified:** `docker/docker-compose.yml`
- **Commit:** c934078

**3. [Rule 2 - Missing Critical Data] test_data rows added to MySQL 03-cross-join.sql**
- **Found during:** Task 2
- **Issue:** The old MySQL `02-data.sql` contained both TPC-H seed data AND `test_data` rows. The rewrite of `02-data.sql` for SF1 loading removes those test_data rows. The plan mentioned creating `03-seed.sql` for PostgreSQL but didn't specify the MySQL equivalent. Without this, `test_mysql.py` data tests would fail (they query `test_data`).
- **Fix:** Extended `docker/init/mysql/03-cross-join.sql` to include `test_data` INSERT IGNORE rows alongside the existing `departments` seed.
- **Files modified:** `docker/init/mysql/03-cross-join.sql`
- **Commit:** c934078

## Known Stubs

**Query expected row counts are estimates based on TPC-H SF1 reference values.** The custom Python generator (using `random.seed(42)`) produces statistically valid but not spec-identical data distributions. The actual row counts returned by some queries (especially those with date-range or percentage filters like Q06, Q14, Q17, Q19) will differ from the standard TPC-H SF1 reference values.

The `-- Expected: N rows` annotations in query files use TPC-H reference values as a starting point. After a full Docker Compose integration run, these annotations should be updated to reflect the actual observed row counts from the custom-generated dataset. The `test_queries.py` test will fail for queries where the annotation doesn't match actual output.

**Affected files:** All 44 `queries/{postgres,mysql}/03-q*.sql` files.
**Resolution:** Run `./run-verify.py` with real StarRocks packages, observe failures, update `-- Expected: N rows` annotations to match actual query results.

Queries with hardcoded result sizes (LIMIT N) are immune: Q02 (LIMIT 100), Q03 (LIMIT 10), Q10 (LIMIT 20), Q18 (LIMIT 100), Q21 (LIMIT 100). Those `Expected:` annotations are exact.
