  <action>
    Remove the four target files. Use `git rm` (NOT `rm -f`) so the deletions are tracked and the change becomes a single staged deletion event in git, ready for commit.

    ```bash
    # Single command — atomic deletion of all four files. -f flag is unnecessary
    # because none of the files are modified in the working tree (clean checkouts).
    git rm lib/docker_backends.py lib/starrocks.py lib/tls.py lib/driver_registry.py
    ```

    Then clear the stale bytecode. `__pycache__/` is in the project's `.gitignore` (verify with `grep '__pycache__' .gitignore` — it is) so it is not tracked by git. Use `rm -rf` (not `git rm`) for the directory:

    ```bash
    rm -rf lib/__pycache__
    ```

    Why delete the whole `__pycache__/` directory rather than just the four `.pyc` files? Three reasons: (1) Python regenerates `.pyc` files on demand for any module that gets imported in the next test run, so we lose nothing; (2) leaving stale `.pyc` files can mask import errors (Python prefers a `.pyc` if it matches a missing `.py`'s mtime in some configurations) — clean slate is safer; (3) one `rm -rf` is simpler than four `rm -f`s and mtime comparisons.

    Do NOT touch:
    - `lib/__init__.py` (empty package marker — required for `from lib.catalog_helpers import ...` to keep working)
    - `lib/catalog_helpers.py` (the LIVE module — used by every test in tests/ and by Phase 3's benchmark module)
    - `CLAUDE.md` (04-03 owns the "Retired" table deletion — touching it here would create a merge conflict with 04-03)
    - `.planning/REQUIREMENTS.md` (04-03 owns FS-SR-07 registration)
    - Anything outside `lib/`

    After the two commands above, immediately verify the working tree state:
    ```bash
    ls -la lib/   # Expect: . / .. / __init__.py / catalog_helpers.py — nothing else
    git status -s lib/   # Expect: 4 lines starting with "D " (deleted, staged)
    ```
