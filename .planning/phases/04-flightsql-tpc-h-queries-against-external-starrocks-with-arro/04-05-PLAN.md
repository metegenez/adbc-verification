<context_summary>
- **Why DuckDB tpch extension:** the hand-rolled generator (Phase 2) is 503 lines and produces non-canonical distributions — verified: it omits the `(N, O)` (returnflag, linestatus) combination, so Q01 returns 3 group-by rows instead of the canonical 4 (RESEARCH.md line 464). DuckDB's `INSTALL tpch; LOAD tpch; CALL dbgen(sf=1)` ships dbgen-equivalent generation as part of the extension; output is deterministic and matches the published TPC-H SF1 row counts exactly.

- **HEADER row contract:** All consumers expect a 1-line header — postgres `\COPY ... HEADER true`, mysql `LOAD DATA ... IGNORE 1 ROWS`, sr-external (plan 04-01) `csv.skip_header='1'`. CANONICAL-SPEC.md lines 99/105 require `HEADER TRUE` and explicitly call out that headerless CSVs would silently shift every column by one row. This plan emits `HEADER TRUE` matching the spec and every consumer.

- **Catalog placeholders:** `queries/tpch/q*.sql` uses literal placeholder substrings of the form `{catalog}.{db}.<table>`. The exact placeholder syntax is fixed in CANONICAL-SPEC.md lines 16-29 — see the `<interfaces>` block in `<context>` for the exact text. The substitution is the canonical loader's responsibility (plan 04-02); this plan only authors the placeholder text.

- **Calibration source:** mysql is the calibration anchor — its 22 queries already pass green in the existing test suite, so substituting catalog/db and re-running against canonical data yields the authoritative `Expected (mysql)` values. Counts may differ from the existing literal values because the canonical generator produces strictly different row distributions than the hand-rolled one. The 5 working postgres queries (q04, q12, q13, q16, q21) get a separate `Expected (postgres)` calibrated against canonical postgres SF1. The flightsql-starrocks counts are written as the same as mysql under the assumption "same source CSVs, same StarRocks SQL semantics" and will be verified end-to-end by plan 04-02 once sr-external is up.

- **Duplicate sets retired:** `queries/mysql/03-q*.sql` (22 files) and `queries/postgres/03-q*.sql` (22 files) get `git rm`-deleted. `queries/mysql/01-select.sql`, `02-join.sql`, `queries/postgres/01-select.sql`, `02-join.sql` are NOT deleted (Phase 1 smoke queries, not TPC-H, picked up by `tests/test_queries.py` via the existing per-directory mechanism that 04-02 keeps for non-canonical backends).
</context_summary>

    ```python
    #!/usr/bin/env python3
    """
    TPC-H SF1 CSV generator using DuckDB's bundled tpch extension.

    Replaces the 503-line hand-rolled generator (Phase 2) with the canonical
    TPC-H dbgen distribution shipped via DuckDB. Output is deterministic for a
    given scale factor.

    Output: docker/data/sf1/<table>.csv (8 files), gitignored (~895 MB).

    Format: comma-delimited, double-quote-enclosed on demand, LF line endings,
    1-line header row. The header row is required by the existing consumers:
      - docker/init/postgres/02-data.sql      (uses HEADER true)
      - docker/init/mysql/02-data.sql         (uses IGNORE 1 ROWS)
      - docker/init/sr-external/02-data.sql   (plan 04-01; uses csv.skip_header='1')

    Usage:
        cd docker && python3 generate-sf1-data.py
    """
    import os
    import duckdb

    SCRIPT_DIR = os.path.dirname(os.path.abspath(__file__))
    OUT_DIR = os.path.join(SCRIPT_DIR, "data", "sf1")
    os.makedirs(OUT_DIR, exist_ok=True)

    con = duckdb.connect()
    con.execute("INSTALL tpch")
    con.execute("LOAD tpch")
    con.execute("CALL dbgen(sf=1)")

    TABLES = ["region", "nation", "supplier", "part", "partsupp",
              "customer", "orders", "lineitem"]
    for table in TABLES:
        out_path = os.path.join(OUT_DIR, f"{table}.csv")
        con.execute(
            f"COPY {table} TO '{out_path}' "
            f"(FORMAT CSV, HEADER TRUE, DELIMITER ',')"
        )
        n = con.execute(f"SELECT count(*) FROM {table}").fetchone()[0]
        print(f"  {table}: {n} rows -> {out_path}")
    ```

    Notes:
    - DuckDB's `COPY ... (FORMAT CSV)` defaults to LF line endings on Linux; verified empirically in DuckDB 1.0+. Task T03 explicitly checks `file <csv>` does not report CRLF.
    - `HEADER TRUE` matches CANONICAL-SPEC.md (corrected) and every existing consumer (postgres `\COPY ... HEADER true`, mysql `IGNORE 1 ROWS`, sr-external `csv.skip_header='1'`).
    - Table order matches existing consumer order (see `docker/init/postgres/02-data.sql`): region, nation, part/supplier (parents), partsupp, customer, orders, lineitem.
    - No `random.seed(42)` needed — DuckDB's tpch is deterministic by spec.
