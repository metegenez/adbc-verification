<objective>
Add the `sr-external` Compose service that runs a second StarRocks instance (FE+BE co-located), reusing the existing `docker/Dockerfile` (image already has Arrow Flight ports 9408/9419 configured per `docker/Dockerfile:21,26`). Populate it with TPC-H SF1 data — a `tpch` database with 8 native StarRocks tables (region, nation, supplier, part, partsupp, customer, orders, lineitem) loaded via the `FILES()` table function from the existing `docker/data/sf1/*.csv` bind-mount. Make `sr-main` wait for `sr-external` to be healthy before starting, and ensure init is idempotent across cold/warm boots so the verification suite can run repeatedly without leaking state. This plan delivers FS-SR-01, FS-SR-02, FS-SR-03 from REQUIREMENTS.md (the IDs introduced by Phase 4) and implements user decisions D-01 through D-08 from `04-CONTEXT.md`. It is the foundation that 04-02 (queries + tests) and 04-03 (run-verify.py + REQUIREMENTS.md sync) build on top of.
</objective>

<context_summary>
- **Reuse, don't reinvent (D-01, D-03):** sr-external uses `build: .` — the same image as sr-main. Arrow Flight ports 9408 (FE) / 9419 (BE) are already in `fe.conf`/`be.conf` (`docker/Dockerfile:21,26`); EXPOSE on `docker/Dockerfile:33`. The `entrypoint.sh` lines 86-94 already loop over `/docker-entrypoint-initdb.d/*.sql` after StarRocks is up — sr-external inherits this for free.
- **Compose-internal only (D-02):** No host ports for sr-external. sr-main reaches it via Docker DNS at `sr-external:9408`. CSV mount path inside the container: `/opt/starrocks/data/sf1/` (parity with the baked-in `/opt/starrocks/data/` set by `Dockerfile:29`; verified live by RESEARCH.md Pattern 1 — `INSERT FROM FILES('file:///opt/starrocks/data/sf1/lineitem.csv'...)` loads 6M rows in ~3.7s).
- **Read-only mount is safe (D-06):** Unlike MySQL's chown-on-startup trap (CLAUDE.md "Volume mount: do NOT add `:ro`"), StarRocks does **not** chown its mounts at runtime. `./data/sf1/:/opt/starrocks/data/sf1/:ro` is verified safe by RESEARCH.md Pitfall 5 ("StarRocks does NOT chown its mounts"). This also enforces "this is reference data" semantics.
- **Idempotency via TRUNCATE+INSERT (D-07, D-08, RESEARCH.md Pattern 3):** Each table's load is `TRUNCATE TABLE tpch.<t>; INSERT INTO tpch.<t> SELECT * FROM FILES(...)`. `CREATE DATABASE IF NOT EXISTS` + `CREATE TABLE IF NOT EXISTS` in the schema file. Verified by research: re-running the init script does not duplicate rows.
- **DDL pitfall avoidance (RESEARCH.md Pitfall 1):** OMIT `DUPLICATE KEY` (StarRocks auto-picks the prefix; explicit composite keys with non-prefix columns error 1064 — verified live). OMIT `BUCKETS N` (auto-tuned since v2.5.7). Use `DISTRIBUTED BY HASH(<first_col>) PROPERTIES('replication_num' = '1')` only — matches single-BE deployment per `docker/Dockerfile:22 (default_replication_num = 1)`.
- **Healthcheck timing (RESEARCH.md Pitfall 3):** Same shape as sr-main healthcheck (`mysql --protocol=TCP -uroot -h127.0.0.1 -P9030 -e "SELECT 1"`) — TCP, not socket (CLAUDE.md "MySQL healthcheck: TCP, not socket"). `start_period: 180s` covers FE+BE warmup (~60s) + SF1 load (~30s with FILES() vs ~120s with mysql LOAD DATA) + 90s buffer for slower hosts.
- **Cross-plan contract:** 04-02 will assume the catalog URI `grpc://sr-external:9408` reaches sr-external, the database `tpch` exists, and all 8 tables have SF1 row counts (region=5, nation=25, supplier=10000, customer=150000, part=200000, partsupp=800000, orders=1500000, lineitem=6001215). 04-03 will assume the service name is exactly `sr-external` for `run-verify.py:_wait_for_healthy()` services dict (RESEARCH.md Pitfall 7).
